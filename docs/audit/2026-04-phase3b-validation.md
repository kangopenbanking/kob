# Phase 3b — Live Connector Validation (BLOCKED)

**Date:** 2026-04-17
**Status:** BLOCKED at platform layer.

## Summary

Phase 3b was unable to complete the end-to-end poll → upsert → audit cycle because the Lovable Cloud edge runtime is **not honoring `verify_jwt = false`** declared in `supabase/config.toml` for the four banking workers.

## Reproduction

1. `supabase/config.toml` contains explicit `[functions.bank-data-poller]` block with `verify_jwt = false` (committed in Phase 3, file present, format valid — confirmed by `code--view`).
2. Code change to `bank-data-poller/index.ts` was made (timestamp comment + console.log diagnostics) to force a clean redeploy. Edge function logs confirm `booted` events occurred after the change, proving the **code** was redeployed.
3. 30 `POST` requests sent over 2.5 minutes to `/functions/v1/bank-data-poller` with the same headers pg_cron sends:
   - `Authorization: Bearer <project anon key>`
   - `x-cron-secret: kob-cron-2026`
   - `Content-Type: application/json`
4. **Every request returned `HTTP 401 {"error":"Unauthorized"}`**.
5. **The diagnostic `console.log('[bank-data-poller] request received', ...)` placed at line 1 of the Deno.serve handler never fired** (verified via `supabase--edge_function_logs` search).

## Conclusion

The 401 response is generated by the platform's JWT verifier **before** the user code runs. The `{"error":"Unauthorized"}` body shape is identical to our in-code `verifyCronAuth` response, which previously masked the root cause. The redeploy completed successfully (booted logs present) but the per-function `verify_jwt = false` setting is being ignored.

This is a **Lovable Cloud platform behavior**, not a code defect. The same pattern works on a stock self-hosted Supabase project, where `supabase functions deploy --no-verify-jwt` or the equivalent `config.toml` block disables the gateway-level JWT check.

## Impact

All Phase 3b validation steps that depended on a successful POST to the worker endpoints could not be executed:
- ❌ `bank_sync_jobs.last_run_at` populated
- ❌ `bank_connector_attempts` row appears
- ❌ Retry queue exercised
- ❌ Reconcile engine exercised against a live config

## Workarounds (none viable from code-only side)

1. **Rotate `CRON_SECRET` env var to known value** — would let manual testing succeed, but the platform 401 happens *before* any in-code secret check, so this does not help.
2. **Use service-role key in Authorization header** — same outcome; the platform rejects before code runs.
3. **Ask Lovable to re-evaluate `verify_jwt` propagation for these four functions** — required.

## Audit Trail

- Standing Order 3 (Audit Trail): root cause is platform-side `verify_jwt` enforcement that contradicts the project-level `config.toml` declaration. Reference: [Supabase Functions §`verify_jwt` config option](https://supabase.com/docs/guides/cli/config#functions.function_name.verify_jwt).
- Standing Order 4 (Surgeon): no code removed; only diagnostic logging added then reverted. `config.toml` blocks remain.
- Standing Order 6 (Version Gate): no API surface change. No version bump required.

## Recommendation

Phase 3b is **paused pending platform-side resolution**. Phase 4 (Payment Gateway audit) and other phases that do not depend on banking-cron execution are unblocked and can proceed.
